Brainstorm Output: Semantic Brain Indexing (CodeGraph v3) ​

Generated by: cm-brainstorm-idea Date: date

Phase 1: DISCOVER ​

The user expressed that the current skeleton.md generation is "too simple" and "if you look at it, it has no value for humans". It only extracts the directory and file names. In contrast, the CodyMaster codebase is a "Brain" consisting of semantic components:

  • skills/: Neocortex (Behavioral patterns, logic gates, workflows like cm-planning, cm-ux-master, etc.)
  • cm-continuity: Hippocampus (Short-term working memory)
  • cm-notebooklm: Cerebrum / Long-term Memory
  • src/ & scripts/: Motor cortex / Executable tools.

Without context, skills/cm-auto-publisher is just a file string. With context, it is "A core sub-agent skill bridging LLMs to Astro".

Phase 2: DEFINE ​

Qualified Problem ​

For: Product Managers, Developers, and Public Viewers exploring CodyMaster. Who: See a dense 3D map of nodes but have no idea what each node actually means. The: Current skeleton.md indexer is a purely structural crawler. That: Generates a filesystem tree devoid of semantic meaning. Unlike: A true "Brain Map" that explains the purpose and thinking model of the codebase. Our approach: Enhance the indexer to parse and inject human-readable metadata, specifically targeting the CodyMaster Skills, so the CodeGraph becomes an interactive encyclopedia of the system's brain.

Phase 3: DEVELOP ​

Option A: AST & Frontmatter Extraction (Lightweight Semantic) ​

Approach: Modify cm-codeintell / index-codebase.sh to parse the YAML frontmatter in SKILL.md files (and Markdown headers) and inline docstrings (/// or /**) from TS/JS code. Inject a desc: "..." field directly into skeleton.md (via a JSON companion file or markdown quotes). Philosophy: Tech-first, deterministic, high speed. Effort: S (~2 hours) Pros:

  • Extremely fast to run locally.
  • 100% accurate (relies on developer's own comments). Cons:
  • Relies on strict documentation discipline. Un-documented files still have no context.

Option B: AI-Powered "Brain Scanner" (Deep Semantic Map) ​

Approach: Use an LLM agent script (cm-dockit variant) to scan the entire codebase locally, reading every file and using an AI prompt to write a 1-sentence "System Role" for each file (e.g. "This file handles the frontend routing", "This is the core Memory module"). Save this mapping into .cm/brain_map.json. Philosophy: User-first (rich storytelling). Effort: M (~1 day) Pros:

  • Guarantees 100% human-readable context.
  • Makes the code graph truly "intelligent". Cons:
  • Requires token expenditure and time to re-index the codebase every major update.

Option C: Hybrid Taxonomy "Lobes & Synapses" Model ​

Approach: Map the project visually and semantically into "Brain Regions" using a predefined taxonomy file. For instance, any file under skills/ is tagged [Frontal Lobe: Decision Making]. Any file in cm-notebooklm is [Hippocampus: Memory]. The Node.js indexer uses regex rules + frontmatter to match files to these Lobes. The 3D CodeGraph clusters them visually or assigns distinct organic colors per Lobe. Philosophy: Business-first / Product-first (Extreme "WOW" factor and storytelling). Effort: M (~1 day) Pros:

  • Delivers the exact "Brain Model" narrative the user wants to communicate.
  • Requires no heavy LLM processing on every index run. Cons:
  • Less granular than Option B (files inside a Lobe share a generic context unless they have specific frontmatter).

🎯 Recommendation ​

Choose Option C: Hybrid Taxonomy "Lobes & Synapses" Model combined with Option A (Frontmatter Extraction).

Why this? By categorizing CodyMaster into definitive brain regions and combining that with Frontmatter yaml extraction of SKILL.md logic files, we achieve both high-level storytelling (Lobes) and granular human readability (Skill descriptions).

Next Steps for cm-planning:

  1. Update the indexer to output a .cm/brain.json that stores node_id, brain_region (skills, core, memory), and description (parsed from Yaml frontmatter).
  2. Write a Node script (scripts/index-brain.mjs) that implements this parsing and taxonomy rules.
  3. Update codegraph.astro to color-code nodes by brain_region and display the parsed description in the right-side info panel, transforming it from a "File Tree" to a "Brain".
