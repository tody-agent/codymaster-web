**For:** AI Agents (CodyMaster) & Developers

## Options Evaluated

### Option A: The "Co-Pilot" Approach (Recommended)
**Approach:** Support BOTH Stitch and Pencil.dev natively by detecting available MCPs.
**Philosophy:** Balanced (Gives user choice between fast exploration and pixel-perfect control).
**What changes:**
- `cm-ui-preview`: Remove "Future" TODO. Add logic to branch execution: use Stitch for quick concepts, use Pencil.dev (via `batch_design`) for structured layout control.
- `cm-design-system`: Update `DESIGN.md` standard to act as context for Pencil's component insertion.
- `cm-ux-master`: Update language across SKILL.md to say "Stitch & Pencil integration".
**Effort:** Medium
**Risk level:** Low

### Option B: The "Pencil First" Approach
**Approach:** Completely replace Stitch with Pencil.dev across the entire design pipeline.
**Philosophy:** Tech-first (Focus entirely on the local editable `.pen` structure).
**What changes:**
- Purge Stitch API references (`mcp_StitchMCP_create_project`, etc.) from docs.
- Rewrite all 3 skills to exclusively use Pencil's `pencil_batch_design` format.
**Effort:** Large
**Risk level:** High (removes Stitch entirely which might break existing workflows if Pencil is not suited for quick concept generation).

---

## Recommendation

**Choose Option A: The "Co-Pilot" Approach**

### Why Option A:
1. **Preserves functionality:** Stitch is great for fast text-to-UI conceptualization, while Pencil gives total precise control (`batch_design`). Having both makes the kit elite.
2. **Fulfills the prompt:** Cleanly "removes the TODO" in `cm-ui-preview` and fully integrates `pencil.dev` into the workflows of `cm-ux-master` and `cm-design-system`.
3. **Graceful degradation:** If a user hasn't installed the Pencil MCP, the agent can still gracefully fall back to Stitch.
