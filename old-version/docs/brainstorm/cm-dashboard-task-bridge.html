Brainstorm Output: cm-dashboard Task Bridge ​

Generated by: cm-brainstorm-idea Date: 2026-03-23

Initiative ​

Upgrade cm-dashboard skill so that tasks from Claude Code (TodoWrite) and Claude Desktop automatically appear on the dashboard — no manual entry.

Also researched: https://github.com/terranc/claude-telegram-bot-bridge

Qualified Problem ​

For: Developers using Cody Master with multiple AI agents (Claude Code, Claude Desktop, etc.) Who: Cannot see real-time task progress across sessions without manually updating the board The: cm-dashboard lacks automatic bridges to Claude Code TodoWrite events and Claude Desktop That: Should reflect live task state from all agents without any manual action Unlike: Current state where the board is a static, manually-curated kanban Our approach: Wire existing hooks + MCP infrastructure to push tasks automatically to the dashboard

Root causes:

  1. PostToolUse hook for TodoWrite is never configured in settings.json
  2. No MCP server exists for Claude Desktop (which has no hooks system)
  3. SKILL.md documents no auto-sync setup instructions

Impact if NOT addressed:

  • Dashboard remains a manual tool, negating the "single pane of glass" promise
  • Developers must duplicate work: track in conversation AND move cards manually
  • Multi-agent coordination is invisible

9 Windows Summary ​

PASTPRESENTFUTURE
Super-systemTodos = local dev habit, no cross-tool syncMulti-agent → tasks live in silosUnified ops across all AI agents expected
SystemDashboard = manual board15s poll, no auto-sync from agentsDashboard = live task hub synced to all sources
Sub-systemTasks created by handauto-sync API exists, hooks not wiredHooks + MCP bridge runs transparently

Options Evaluated ​

Option A: Hooks-Only Bridge (Claude Code) ​

  • PostToolUse hook → shell/JS script → POST /api/tasks/auto-sync
  • Effort: S (~2-3h) | Risk: Low
  • Score: 8.0/10

Option B: MCP Server Bridge (Universal) ​

  • New MCP server wrapping dashboard API, mounted in both CC and Desktop
  • Effort: M (~6-8h) | Risk: Medium
  • Score: 7.6/10

Option C: Telegram-Style Daemon ​

  • Background daemon parsing subprocess stdout, inspired by claude-telegram-bot-bridge
  • Effort: XL (~2-3 days) | Risk: High (fragile)
  • Score: 5.1/10

Recommendation ​

Hybrid A+B — implement in sequence:

  1. Phase 1 (Ship first): Hooks bridge for Claude Code

    • New scripts/todo-bridge.js
    • Settings.json hook config: matcher=TodoWrite, PostToolUse
    • Reuses existing /api/tasks/auto-sync endpoint
    • Auto-detects projectId from session working directory

  2. Phase 2 (Claude Desktop): MCP server

    • New src/mcp-bridge.ts
    • Tools: sync_tasks(todos[]), update_task(id, status), get_tasks(projectId)
    • Claude Desktop mounts via claude_desktop_config.json

Key Technical Insights from Telegram Bot Research ​

The claude-telegram-bot-bridge uses:

  • Python subprocess → Claude CLI → long-poll pattern
  • SESSION_ROOT sandbox for file access
  • launchd integration for daemon persistence

Applicable to future work: If we ever want remote dashboard control from mobile/Telegram, this is the pattern. Not needed for current task sync initiative.

Next Steps for cm-planning ​

  1. Scope: Build todo-bridge.js + settings.json hook config + SKILL.md upgrade
  2. Constraints:
    • Must reuse /api/tasks/auto-sync — no new endpoints needed
    • Bridge script must be installable to ~/.claude/ (not project-specific)
    • No breaking changes to current dashboard UI or API
  3. Technical decisions already made: D005 in decisions.json
  4. Open questions:
    • Bridge script in JS (Node) or shell? JS preferred (easier JSON parsing)
    • MCP server — separate npm package or bundled in dist/?
    • Should TodoWrite deletions propagate to dashboard?

Context for downstream skills ​

  • cm-planning: Focus on Phase 1 first (hooks bridge). Plan 3 files: todo-bridge.js, settings-hook-snippet.json, updated SKILL.md
  • cm-tdd: Test that auto-sync endpoint correctly upserts by conversationId and maps todo status to task columns
  • cm-execution: Each phase is independent — can be dispatched as separate worktree tasks
