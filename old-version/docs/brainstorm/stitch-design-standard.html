Brainstorm Output: Standardizing Stitch DESIGN.md via cm-ux-master ​

Generated by: cm-brainstorm-idea Date: 2026-03-22

Qualified Problem ​

For: AI Agents (Cody Master) and Frontend Developers Who: Need to generate pixel-perfect, consistent UIs using Google Stitch The: Current cm-ux-master extraction (Harvester v4) That: Generates comprehensive UI tokens and design rules, but lacks a native bridge to Stitch's required engine format Unlike: Manual transcription or vague prompt-based design generation Our approach: Standardize the DESIGN.md format (Markdown + JSON Tokens) as the universal output of cm-ux-master and the mandatory input for cm-ui-preview (Stitch integration).

Root Cause(s): ​

  1. [Technical] Stitch requires a specific Markdown structure (Overview, Colors, Typography) and a hidden <!-- STITCH_TOKENS_START --> JSON block to strictly enforce standard tokens.
  2. [Product] cm-ux-master Harvester extracts immense detail but outputs custom unstructured formats or generic tailwind configs that Stitch's AI engine cannot directly absorb.

Impact if NOT addressed: ​

  • Stitch generates inconsistent UIs because it lacks rigid structural tokens.
  • We lose the power of Harvester v4's visual extraction because the extracted UI cannot be instantly cloned/generated by Stitch.

9 Windows Analysis ​

PASTPRESENTFUTURE
SUPER-SYSTEMManual design handoffs (Figma to Code).AI tools generating UIs based on text prompts (hit or miss).Seamless cloning: Website URL → Exact UI Replica via AI.
SYSTEMcm-ux-master and cm-ui-preview operating as separate conceptual silos.Harvester extracts tokens; Stitch generates UI, but they speak different languages.DESIGN.md acts as the universal standard API between extraction and generation.
SUB-SYSTEMColors, fonts, and spacing defined ad-hoc in CSS.Harvester outputs massive JSON; Stitch wants Markdown + specific JSON structure.Harvester automatically transpiles and outputs Stitch-compliant DESIGN.md.

Options Evaluated ​

Option A: Create a translation script (harvester_to_stitch.py) ​

Approach: Build a middleware script that converts Harvester's raw JSON into DESIGN.md. Pros: Keeps Harvester untouched. Cons: Adds friction. The user or agent has to run an extra step before Stitch can be used.

Option B: Update cm-ui-preview to handle Harvester JSON directly ​

Approach: Teach the Stitch orchestrator skill to read Harvester JSON and parse it into Stitch prompts. Pros: Easy to implement via prompt engineering. Cons: Stitch's AI engine natively expects the DESIGN.md with the STITCH_TOKENS_START block for best results. Bypassing this leads to lower quality.

Option C: Embed DESIGN.md as the Native Output Standard in cm-ux-master ​

Approach: Make the Stitch-compliant DESIGN.md the official standard output of any cm-ux-master harvesting or design system creation. It becomes the source of truth for the entire Cody Master ecosystem. Pros:

  • "One file to rule them all."
  • Direct copy-pasting of a website into a Stitch-ready format.
  • Seamless integration with cm-ui-preview. Cons: Requires updating the core cm-ux-master skill instructions.

Recommendation ​

Choose Option C: Embed DESIGN.md as the Native Output Standard

Why Option C: ​

  1. Frictionless Workflow: A user can scan a website and immediately use Stitch to clone/modify it without manual translation.
  2. Native Stitch Alignment: By writing the exact JSON token block Stitch expects (<!-- STITCH_TOKENS_START -->), we guarantee maximum UI consistency and quality.
  3. Universality: It fulfills the user's objective to make this file the "common standard" for all design representation in the project.

Next Steps for cm-planning: ​

  1. Update cm-ux-master/SKILL.md to include instructions for outputting the DESIGN.md standard.
  2. Update cm-ui-preview/SKILL.md to rigidly read and enforce the DESIGN.md standard before calling Stitch MCP.
  3. Create a template/reference file skills/cm-ux-master/DESIGN_STANDARD_TEMPLATE.md to serve as the definitive guide for AI agents.
