Brainstorm Output: Home Page Redesign (Aligned with /skills) ​

Generated by: cm-brainstorm-idea

Qualified Problem ​

For: New users, founders, and developers visiting the CodyMaster landing page. Who: Need to quickly understand the value of the kit. The: index.html landing page is the first touchpoint That: needs to visually match the sleek, modern, app-like experience of the /skills library. Unlike: The current page, which uses a traditional, somewhat fragmented, long-form landing page design with bespoke CSS (home.css). Our approach: Unify the visual language by inheriting the Bento Grid design, cohesive cards, and minimalist typography from /skills.

Root Cause(s): ​

  1. index.html evolved organically with bespoke classes (home-hero, home-section).
  2. skills.html was built with a stricter, cleaner design system (kit.css, skills-bento).
  3. Disconnect between the "marketing" feel of the home page and the "app/product" feel of the skills page.

Impact if NOT addressed: ​

  • Inconsistent branding (looks like two different products).
  • Missed opportunity to impress users with a cohesive, modern UI right from the start.

9 Windows Analysis ​

PASTPRESENTFUTURE
SUPER-SYSTEM (Market)Landing pages were long-form text blocks.Users scan quickly; they prefer app-like, dashboard interfaces (bento grids).Everything is an interactive, spatial layout.
SYSTEM (Product)The home page was a basic summary of the repo.The home page has custom sections (problem, solution, timeline).The home page feels like an extension of the tool itself—clean, grid-based, app-like.
SUB-SYSTEM (Components)Bespoke CSS classes for every section block.Mixed styles (home-team-card, home-feature-block, home-wf-step).Unified component language (reusable cards, unified typography, identical hover states).

Options Evaluated ​

Option A: The Full Bento Grid Re-architecture ​

Approach: Convert the entire structural flow of the landing page into a massive, interlocking Bento Grid layout (similar to Apple's product pages or the /skills grid), replacing the traditional horizontal "striped" sections. Philosophy: Design-first / App-first What changes:

  • Remove .home-section vertical padding paradigm.
  • Pack "The Problem", "The Solution", "Features", and "Workflow" into differently sized cards within a single or dual massive skills-bento style grid. Effort: L (~3-4 hours) | Risk level: Medium (Requires rewriting copy to fit cards) Pros: Extremely modern, maximum consistency with /skills, high "wow" factor. Cons: Marketing narrative might be harder to control if users just scan a grid.

Option B: Unified Component Styling (The Reskin) ​

Approach: Keep the current narrative sections (Hero → Problem → Solution → Features → Workflow), but relentlessly update their internal components to perfectly match the skill-card UI. Philosophy: Balanced What changes:

  • .home-hero updated to match .skills-hero minimalism.
  • .home-feature-block and .home-team-card updated to use the exact var(--bg-card), borders, hover states, and typography as .skills-bento items.
  • Retain the linear storytelling but upgrade the UI fidelity. Effort: M (~1-2 hours) | Risk level: Low Pros: Keeps the proven marketing flow; fast to implement; removes home.css bloat. Cons: Still feels like a traditional landing page, just prettier.

Option C: Interactive Modals Integration ​

Approach: Turn the home page into a mini-app. Make the "Features" and "The Problem/Solution" interactive cards that open up detailed Modals, identical to how clicking a skill opens the skillModal. Philosophy: Experiential What changes:

  • Reduce text on the home page significantly.
  • Introduce home-modal (reusing skill-modal logic) to explain "How it works" or "Why Us" when clicked. Effort: XL (~5 hours) | Risk level: High Pros: Showcases the app's interactive nature immediately. Cons: High friction. Users might not click to read important marketing copy.

🎯 Recommendation ​

Choose Option B (Unified Component Styling) with strong elements of Option A.

Why Option B + A: ​

  1. Preserves Narrative: We need to keep the "Problem -> Solution -> Features" flow because it converts well and educates the user.
  2. Visual Consistency: By styling every card and block exactly like skills-bento cards, the site will feel incredibly unified.
  3. Low Friction/High Impact: We can achieve the sleek /skills look by refactoring home.css to use the same background gradients, grid gaps, hover transformations, and border-radii as kit.css, without having to invent a complex new masonry layout.

Why NOT the other options: ​

  • Option A (Pure Bento) forces us to chop up our carefully written marketing copy to fit into small boxes, potentially losing the persuasive argument.
  • Option C (Modals) hides too much information behind clicks, increasing bounce rates for a landing page.

Quick-win suggestion: ​

Start by refactoring the .home-team-grid, .home-feature-list, and .home-problem-card to look exactly like the skill cards (same background, border, hover animation, and icon placement). Then tighten the Hero section.

Next Steps for cm-planning: ​

  1. Scope: Refactor index.html and css/home.css to align with the visual language of skills.html and css/kit.css.
  2. Key constraints: Do not remove the core marketing copy; just re-wrap it in better UI components. Keep the Dashboard iframe section intact as it's a strong visual anchor.
  3. Technical decisions: Rely more on kit.css variables (--bg-card, --border-subtle, --radius-xl) and standard typography. Remove redundant bespoke styles from home.css.
  4. Actionable Tasks:
    • Update Hero typography/spacing to match skills hero.
    • Refactor Team items, Feature items, and Problem items into standard "bento-style" cards with matching hover states.
    • Refactor the "How It Works" workflow to feel more polished, perhaps integrating standard card backgrounds.
